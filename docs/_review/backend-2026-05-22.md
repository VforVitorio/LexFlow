# LexFlow backend review — 2026-05-22

## Executive summary

- **Error contract diverges from CLAUDE.md §6.** Custom handlers return `{error, message}`, but the contract says `{detail}`. The frontend error boundary will not match.
- **Sync I/O on async path.** `versions.py` uses `def` for handlers calling blocking `subprocess.run` — that part is fine, but `articles.list_articles` parses laws synchronously inside an `async def`, blocking the event loop on first hit of a multi-MB file.
- **Module-level mutable singleton in `graph.py`** (`_cached_graph: LegalGraph | None`) — racy under Uvicorn workers and not overridable in tests via `dependency_overrides`.
- **`/search` is mounted at the root**, not under `/laws` — three different prefix conventions coexist (`/laws`, `/graph`, `/search`). Documented as `/api/v1/search` only, no resource grouping.
- **MCP tools duplicate route logic** (`get_law`, `get_article`, `search_law`) with their own ad-hoc error envelopes; any future change to the API model has to be made twice.

## Findings

### 1. Error envelope contradicts the documented contract
**Severity:** high
**Where:** `src/lexflow/api/error_handlers.py:21-45`
**What:** Handlers return `{"error": "<Code>", "message": "<msg>"}`. CLAUDE.md §6 says FastAPI default `{"detail": "<message>"}`; the frontend error boundary reads `detail`.
**Why it matters:** Every 4xx/5xx breaks the React error toast; SDK type generation will also be off.
**Suggested fix:** Either raise `HTTPException(status_code, detail=str(exc))` from a single handler, or rewrite the four handlers to emit `{"detail": ..., "code": ...}` and document it. The `graph.py` path endpoint already does it the right way.
**Suggested issue title:** `[Refactor]: Align error envelope with documented {detail} contract`

### 2. Sync file parse inside async handler
**Severity:** high
**Where:** `src/lexflow/api/routers/articles.py:21`, `laws.py:44`
**What:** `list_articles` / `get_law` are `async def` but call `registry.get_law(law_id)` which performs blocking `Path.read_text` + regex parse on first access. For an unparsed law, a single request blocks the loop for hundreds of ms.
**Why it matters:** A handful of concurrent cold requests freezes the event loop; defeats the point of FastAPI async.
**Suggested fix:** Either make those handlers `def` (FastAPI runs sync routes on a threadpool — versions.py already does this), or wrap the call in `asyncio.to_thread`.
**Suggested issue title:** `[Refactor]: Run blocking parser via threadpool or convert handlers to sync def`

### 3. Module-level graph cache bypasses DI
**Severity:** high
**Where:** `src/lexflow/api/routers/graph.py:26-34`
**What:** `_cached_graph: LegalGraph | None = None` plus an unsynchronized `if None: build` inside `get_graph_dep`. Already a third caching layer (the other two are `graph.model.get_graph` and `graph.cache.load_or_build`).
**Why it matters:** (a) race condition under Uvicorn `--workers >1` and even within one process under starlette's threadpool; (b) test fixtures cannot override it via `app.dependency_overrides`; (c) duplicates the `lru_cache` strategy used everywhere else.
**Suggested fix:** Either reuse `lexflow.graph.model.get_graph` (already an `lru_cache`) or move the singleton into a dedicated `graph_provider` module with a `Lock`. Stop the `global` mutation.
**Suggested issue title:** `[Refactor]: Replace module-global graph cache with lru_cache provider`

### 4. Duplicate graph singletons across three modules
**Severity:** medium
**Where:** `graph/model.py:69` (`get_graph`), `graph/cache.py:57` (`load_or_build`), `api/routers/graph.py:26` (`_cached_graph`)
**What:** Three different "give me the singleton graph" entry points, none of them used by the live API (the API ignores `cache.load_or_build`, so the on-disk cache is dead code at runtime).
**Why it matters:** A non-obvious "which one is real" question for any contributor; the JSON cache silently does nothing.
**Suggested fix:** Pick one — `graph.cache.load_or_build` is the most useful (persistent, hash-validated). Have `dependencies.get_graph` call it and delete the other two.
**Suggested issue title:** `[Refactor]: Consolidate graph singleton + delete dead JSON cache path`

### 5. `LegalGraph._g` accessed from outside the class
**Severity:** medium
**Where:** `src/lexflow/graph/cache.py:50` (`graph._g = g`)
**What:** Cache loader mutates a private attribute directly. Combined with `LegalGraph.graph` property leaking the raw `DiGraph` to callers (`graph.py:81-86`), encapsulation is essentially nominal.
**Why it matters:** Any future change to `LegalGraph` internals — e.g. wrapping in Neo4j — breaks two unrelated modules.
**Suggested fix:** Add `LegalGraph.from_networkx(g)` constructor; expose only read-only methods (`density()`, `weakly_connected_components()`) rather than the raw graph.
**Suggested issue title:** `[Refactor]: Encapsulate LegalGraph, remove raw DiGraph leakage`

### 6. `/search` is not nested as a sub-resource
**Severity:** medium
**Where:** `src/lexflow/api/routers/search.py:11-15`, `app.py:41`
**What:** Mounted as `/api/v1/search`, not `/api/v1/laws/search`. Adjacent endpoints under `/laws/{id}/...` follow REST nesting; search does not.
**Why it matters:** Inconsistent shape for the generated TS client; future `/articles/search` or `/graph/search` will multiply the inconsistency.
**Suggested fix:** Move to `/api/v1/laws/search` (it's a search *over laws*). Keep the alias deprecated only if anything ships before v1 freezes.
**Suggested issue title:** `[Refactor]: Move search under /laws prefix for consistent resource nesting`

### 7. Versions endpoints not under v1, sync handlers
**Severity:** low
**Where:** `src/lexflow/api/routers/versions.py:27,43`
**What:** `def list_versions`/`get_diff` (sync) is correct given subprocess; but they live at `/api/v1/laws/{law_id}/versions` *and* `/diff`. The `/diff` path takes two arbitrary commit hashes via `?from=&to=` — no validation that they look like SHAs before being passed to `git diff`.
**Why it matters:** Even with `subprocess.run` shell-disabled, accepting arbitrary refs lets a caller probe arbitrary objects in the legalize-es submodule (e.g. `HEAD~1000`). Not a CVE, but smells.
**Suggested fix:** Add a `^[0-9a-f]{7,40}$` regex on `from_commit`/`to_commit` query params via `pattern=`.
**Suggested issue title:** `[Refactor]: Validate commit hash params in /versions and /diff`

### 8. `PaginationParams` is a class with no behavior
**Severity:** low
**Where:** `src/lexflow/api/dependencies.py:15-27`
**What:** Three lines of stored state, no methods. Per CLEAN_CODE this should be a dataclass or a Pydantic model + `Annotated[..., Depends()]`.
**Why it matters:** Tiny style nit, but FastAPI's modern pattern is `Annotated[PaginationParams, Depends()]` with `PaginationParams` as a Pydantic v2 model (gets free validation docs in the OpenAPI schema).
**Suggested fix:** Convert to `class PaginationParams(BaseModel)` with `model_config`, used via `Annotated[PaginationParams, Depends()]`. Same treatment for `_get_git_reader`.
**Suggested issue title:** `[Refactor]: Adopt Annotated[Depends()] + Pydantic for dependency types`

### 9. Config uses dataclass + os.environ, not pydantic-settings
**Severity:** low
**Where:** `src/lexflow/utils/config.py:14-47`
**What:** `pydantic-settings>=2.7` is in `dependencies` but unused. A frozen dataclass reads `os.environ.get(...)` manually and silently uses `int(..., 20)` — non-int env var raises `ValueError` at first call, not at startup.
**Why it matters:** No `.env` support, no validation at boundary, dep installed for nothing.
**Suggested fix:** Replace with a `BaseSettings` subclass (`env_prefix="LEXFLOW_"`). Crash loudly on misconfig.
**Suggested issue title:** `[Refactor]: Migrate utils/config.py to pydantic-settings`

### 10. MCP tools duplicate REST endpoint logic
**Severity:** low
**Where:** `src/lexflow/chat/mcp_server.py:22-87`
**What:** Each tool reimplements the registry lookup and the "not found" branch with a different error shape (`{"error": "not_found", ...}` vs HTTP 404). No shared helper.
**Why it matters:** Any rename of a field in `Law` / `Article` requires editing both layers; the agent gets a different contract than the React client.
**Suggested fix:** Extract a `lexflow.core.services` module with `find_law`, `find_article`, `search` that both layers call, then keep the MCP layer just for serialization.
**Suggested issue title:** `[Refactor]: Extract core services to share logic between REST and MCP`

### 11. No CORS / no SPA static mount despite §6 plan
**Severity:** low
**Where:** `src/lexflow/api/app.py:30-42`
**What:** No `CORSMiddleware`, no `StaticFiles` mount, no SPA catch-all. Frontend doesn't exist yet, but the contract is documented.
**Why it matters:** Will be the very first dev-mode 404 the React work hits — best to land it when the contract is written, not on the day someone wires Vite.
**Suggested fix:** Add a small `setup_frontend(app)` helper in `api/app.py` guarded by `frontend/dist` existence. No CORS needed if Vite proxies, but a dev-only `localhost:5173` allow-origin is cheap.
**Suggested issue title:** `[Feature]: Mount frontend dist + SPA fallback in FastAPI app factory`

### 12. `lifespan` does no preloading; first request pays full parse cost
**Severity:** low
**Where:** `src/lexflow/api/app.py:19-27`
**What:** Comment says "Metadata preload is deferred to first request". `LawRegistry.preload_all_metadata()` exists but is never called.
**Why it matters:** First `/laws` request parses YAML for ~12k files synchronously inside the handler. With 12k files this is several seconds of "the API is hanging".
**Suggested fix:** In `lifespan` startup, spawn `asyncio.create_task(asyncio.to_thread(get_registry().preload_all_metadata))`. Production-only path; tests already use a tiny fixture so cost is zero.
**Suggested issue title:** `[Feature]: Preload metadata in lifespan startup task`

### 13. Dashboards iterate full registry every call
**Severity:** low
**Where:** `src/lexflow/dashboards/analytics.py:13-48`
**What:** `reforms_by_year`, `rank_distribution`, `status_distribution`, `jurisdiction_heatmap` all walk every `law_id` and re-parse metadata. No caching, no shared "summary table".
**Why it matters:** Each dashboard view = 12k metadata parses; ironically four near-duplicate functions could be one `def summarize(registry) -> SummaryTable` that everyone reads from.
**Suggested fix:** Add a single `compute_law_summary_table(registry) -> list[LawSummary]` (using `registry._build_summaries`, which is already private), cache it, derive all four plots from it.
**Suggested issue title:** `[Refactor]: Cache law summary table and reuse across dashboards`

### 14. Anthropic model IDs are stale
**Severity:** low
**Where:** `src/lexflow/chat/providers/anthropic_provider.py:12-16`
**What:** Hardcoded list contains `claude-opus-4-6`, `claude-sonnet-4-6`, `claude-haiku-4-5-20251001`. Anthropic Opus/Sonnet 4.7 are GA now; the cutoff (2026-01) postdates them.
**Why it matters:** Anyone selecting the latest Opus from the UI gets a 404 from the API.
**Suggested fix:** Either hit `client.models.list()` like OpenAI provider, or update the constants to 4.7 family.
**Suggested issue title:** `[Refactor]: Refresh Anthropic provider model list to 4.7`

### 15. No tests for graph router, versions, MCP tools, dashboards, providers
**Severity:** medium
**Where:** `tests/` (compared with `src/lexflow/api/routers/graph.py`, `versions.py`, `chat/`, `dashboards/`)
**What:** Tests exist for parser, file_discovery, models, search, health, and three router files. Nothing covers graph endpoints, version endpoints, the MCP server tools, or any chat provider.
**Why it matters:** Graph router has the trickiest singleton and the only `HTTPException` usage — the riskiest code is the least tested.
**Suggested fix:** Add `test_graph_endpoints.py` (using `mock_registry` fixture + a built graph), `test_version_endpoints.py` (with a temp git repo via `subprocess.run(["git", "init", ...])`), and contract tests for one provider with HTTP stubs (`respx`).
**Suggested issue title:** `[Tests]: Cover graph + version routers and MCP tools`

## Cross-cutting recommendations

- **Adopt `Annotated[T, Depends(...)]` everywhere.** The graph router already started; the rest of the routers still use the bare-default signature `registry: LawRegistry = Depends(get_law_registry)`. Pick one style and apply it via `code-refactor`.
- **Introduce a `core/services/` layer** between routers/MCP tools and the registry — the routers should be 5-line orchestrators, not branch logic. Today `articles.py` and `mcp_server.py` both reimplement "find article by number".
- **One error envelope.** Decide between FastAPI default `{detail}` and a richer `{detail, code}`; document in `CLAUDE.md`; make the exception handlers emit it; delete `schemas.ErrorResponse` if unused.
- **Add structured logging at module seams** (registry parse, graph build, git ops). Today only "X built: N items" lines exist; failures use `logger.warning(..., exc_info=True)` in one place and silent `except Exception` (no log) in `graph/cache.py:27, 52` and `builder.py:21, 37`. Either log + reraise, or stop swallowing.
- **Replace `_safe_enum` silent fallback** in `parser.py:63` with a logged warning — corrupt frontmatter currently downgrades to `LawRank.OTRO` invisibly.

## What did NOT need fixing

- **Pydantic v2 usage is clean.** All models use `ConfigDict(frozen=True)`, `Field(...)`, `computed_field`, `field_validator` correctly. No v1 leftovers (`Config` class, `@validator`, `.dict()`).
- **`LawRegistry` lazy parse + double-checked lock** in `_ensure_parsed` is correct and the cleanest piece of the codebase. Tests prove it via `preload_all_metadata`.
- **Module sizes are within budget.** Largest module is `parser.py` at ~360 lines but each helper is < 20 lines and the docstrings explain the recursion. No 600-line god-module.
